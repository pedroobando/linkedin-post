# Product Requirements Documents (PRDs)

Este directorio contiene los Product Requirements Documents del proyecto.

## ¿Qué es un PRD?

Un PRD (Product Requirements Document) define QUÉ se va a construir antes de empezar a desarrollar. Sirve para:

- Clarificar el alcance de una feature
- Definir criterios de aceptación
- Evitar "scope creep" durante el desarrollo
- Documentar el propósito para referencia futura

## Cuándo crear un PRD

Crear un PRD cuando:
- Nueva feature toma más de 2 horas de trabajo
- Cambio afecta múltiples archivos
- Integración con API externa
- Feature que podrías olvidar cómo funciona en 3 meses

## Formato

Usar el formato: `{nombre-feature}.md`

Ejemplos:
- `scheduling-system.md`
- `linkedin-api-integration.md`
- `analytics-dashboard.md`

## Template

```markdown
# Feature: [Nombre de la feature]

## Problem Statement
[¿Qué problema estamos resolviendo?]

## Proposed Solution
[Descripción de la solución propuesta]

## User Stories
- Como [rol] quiero [acción] para [beneficio]

## Acceptance Criteria
- [ ] Criterio 1
- [ ] Criterio 2
- [ ] Criterio 3

## Technical Considerations
[Consideraciones técnicas relevantes]

## Out of Scope
[Lo que NO incluye esta feature]
```
